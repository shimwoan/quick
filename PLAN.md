# 퀵서비스 콜 기반 경로 추천 서비스 기획서

## 1. 서비스 개요

### 1.1 목적
퀵서비스 기사가 이동 중 더 많은 오더를 수주할 수 있도록, **콜 발생 확률이 높은 행정동**을 경유하는 경로를 추천하는 서비스.

### 1.2 서비스 범위 및 전제
- **서비스 지역**: 서울특별시 (행정동 기준 약 467개동)
- **서비스 역할**: 경로 추천 전용. 오더 수락/배차는 기사가 사용하는 별도 앱에서 처리
- **본 서비스의 입력**: 기사가 오더 정보(출발지/도착지)를 직접 입력하거나, 오더를 추가 등록
- **본 서비스의 출력**: 콜 기대지수 높은 행정동을 경유하는 추천 경로 리스트

### 1.3 기존 네비게이션과의 차이

| 구분 | 일반 네비게이션 | 본 서비스 |
|------|----------------|-----------|
| 기준 | 최단거리 / 최단시간 | 콜 밀집 행정동 경유 |
| 목적 | 빠른 도착 | 이동 중 추가 오더 수주 극대화 |
| 경로 특성 | 직선적 | 콜 핫스팟 경유 (약간의 우회 허용) |

## 2. 핵심 기능

### 2.1 사용자 플로우

```
① 기사가 본 앱 실행 (별도 오더 수락 앱에서 오더를 이미 받은 상태)
② 오더 정보 입력 (출발지/도착지 직접 입력 또는 주소 검색)
③ [경로 추천 요청] 버튼 클릭
④ 최단경로 + 추천경로 리스트 + 주변 콜 다발 행정동 리스트 표시
⑤ 기사가 원하는 경로 선택 → 네비게이션 시작
⑥ 이동 중 추가 오더 발생 시 → [오더 추가] 버튼으로 신규 오더 입력
⑦ 현재 위치 기준 경로 재계산 → 새 추천 리스트 표시
⑧ 기사가 경로 재선택 → 네비게이션 계속
```

- 모든 경로 추천/재계산은 **기사가 직접 트리거** (자동 재추천 없음)
- 오더 정보는 기사가 직접 입력 (별도 오더 앱과의 연동은 현재 범위 외)

### 2.2 콜 기반 다중 경로 추천
- 출발지 → 도착지 경로를 **복수 개** 생성하여 기사가 선택할 수 있도록 제공
- **최단경로**: 우회 없는 기본 최단시간 경로 (기준선)
- **추천 1, 추천 2, 추천 3 ...**: 콜 다발 행정동을 경유하는 경로들 (콜 기대지수 높은 순 정렬)
- 각 추천 경로마다 최단경로 대비 **추가 소요시간(+N분)** 을 명시하여 비교 가능하게 제공
- 기사가 시간 여유와 콜 기대치를 보고 직접 경로를 선택

### 2.3 콜 다발 행정동 리스트 제공
- 출발지~도착지 경로 주변 인접동 중 콜 발생 확률 높은 행정동 리스트를 아웃풋으로 제공
- 각 행정동별 콜 기대지수 표시 (시설 밀집도 기반 추정값, 실제 콜 건수가 아님을 UI에 명시)
- 각 행정동을 경유할 경우의 추가 소요시간도 함께 표시

### 2.4 실시간 위치 추적
- 기사 앱에서 GPS 위치를 **주기적으로 서버에 전송** (5초 간격)
- 서버는 항상 기사의 최신 위치 + 진행 방향(heading)을 보유
- 이 실시간 위치가 경로 재계산, 다중 오더 정렬, 전진 방향 판단의 기준점이 됨
- 위치 업데이트 시 현재 어느 행정동에 위치하는지도 함께 판별

### 2.5 다중 오더 경로 최적화
- 진행 중 오더가 추가되면, **실시간 현재 위치 기준**으로 기존 오더 + 신규 오더를 종합하여 경로 재계산
- **핵심 제약: 경로는 항상 전진 방향. 새 오더 추가로 인해 뒤로 되돌아가지 않음**
- 현재 위치에서 남은 픽업/드롭 포인트를 전진 방향으로 정렬 후, 그 사이사이에 콜 다발 행정동 경유
- 이미 완료한 픽업/드롭 포인트는 경로 계산에서 제외

## 3. 콜 발생 확률 스코어링 모델

### 3.1 데이터 소스

| 데이터 | 출처 | 용도 |
|--------|------|------|
| 상권 정보 (업종별 사업체 수) | 소상공인시장진흥공단 상가(상권)정보 API | 서울 행정동별 업종 밀집도 산출 |
| 행정동 경계 폴리곤 | 국토교통부 행정구역 API / 통계청 SGIS | 지리 연산, 인접동 판별 |
| 물류센터/공단 위치 | 한국산업단지공단 API, 공공데이터포털 | 물류 허브 스코어 가산 |
| 병원/의료기관 | 건강보험심사평가원 API | 의료기관 밀집도 |
| 대형 쇼핑몰/유통시설 | 소상공인 상가정보 + 대형유통업체 데이터 | 유통 허브 스코어 가산 |
| 오피스/업무시설 밀집도 | 소상공인 상가정보 (업종 분류) | 사무실 밀집 스코어 |

### 3.2 행정동 스코어 산출 공식

```
행정동 콜 기대지수 = Σ (카테고리별 가중치 × 해당 카테고리 시설 수 / 행정동 면적)
```

#### 카테고리별 가중치 (초기값, 튜닝 필요)

| 카테고리 | 가중치 | 근거 |
|----------|--------|------|
| 물류센터/공단 | 5.0 | 퀵서비스 콜 최다 발생지 |
| 오피스/사무실 밀집 | 4.0 | 서류, 소화물 콜 다수 |
| 병원/의료기관 | 3.5 | 의료 서류/검체 배송 수요 |
| 쇼핑몰/유통시설 | 3.0 | 상품 배송 수요 |
| 일반 소상공인 상가 | 1.0 | 기본 수요 |

#### 스코어 정규화
- 전체 행정동 스코어를 0~100 범위로 정규화

### 3.3 스코어 데이터 관리
- 행정동별 스코어는 **배치로 사전 계산** (일 1회 또는 주 1회 갱신)

## 4. 경로 추천 알고리즘

### 4.1 단일 오더 다중 경로 생성

```
입력: 출발지(위경도), 도착지(위경도)
출력: 최단경로 1개 + 추천경로 N개, 각 경로별 소요시간 비교, 경유 행정동 리스트

1. 출발지 → 도착지 최단경로 산출 (기준선, base_time)
2. 최단경로의 버퍼 영역(좌우 N km) 내 행정동 추출
3. 버퍼 내 행정동을 콜 기대지수 기준 내림차순 정렬
4. 상위 행정동을 경유지 후보로 선정
5. 각 경유지 조합별로 경로를 생성하여 추천 경로 후보 리스트 구성
   - 추천 1: 콜 기대지수 합산 최고 경유지 조합
   - 추천 2: 차선 경유지 조합
   - 추천 3: ...
6. 각 추천 경로에 대해 최단경로 대비 추가 소요시간(+N분) 계산
   - extra_time = 추천경로 소요시간 - base_time
7. 최단경로 + 추천경로 리스트 + 경로 주변 콜 다발 행정동 리스트 반환
```

### 4.2 다중 오더 경로 생성

```
입력: 실시간 현재 위치(GPS), 진행 방향(heading), 진행중 오더들, 신규 오더
출력: 최단경로 + 추천경로 리스트, 각 경로별 소요시간 비교, 경유 추천 행정동 리스트

1. 서버에 저장된 실시간 GPS 위치 + heading을 기준점으로 사용
2. 이미 완료한 픽업/드롭 포인트를 필터링 (현재 위치 이전 + status=completed)
3. 남은 모든 오더의 픽업/드롭 포인트를 진행 방향으로 투영하여 순서 정렬
   - 이미 지나친 방향의 포인트는 경로 뒤쪽이 아닌 전방 우회 경로로 처리
4. 정렬된 포인트 간 구간마다 4.1 알고리즘 적용 (다중 경로 생성)
5. 전체 경로 병합 → 최단경로 + 추천경로 리스트 + 콜 다발 행정동 리스트 반환
6. 각 추천 경로별 최단경로 대비 추가 소요시간(+N분) 계산

제약조건:
- 현재 실시간 위치 기준 후방으로 되돌아가지 않음
- 신규 오더 추가 시 기존 경로의 이미 지나간 구간은 변경 불가
- 현재 위치 → 가장 가까운 다음 포인트부터 경로 시작
```

### 4.3 인접동 판별 로직
- 행정동 폴리곤 경계가 접하거나 중심점 간 거리 기준 (예: 2km 이내)
- 사전에 인접동 매핑 테이블을 생성하여 조회 성능 확보

## 6. API 설계 (초안)

### 6.1 기사 위치 업데이트 (실시간 - Supabase Realtime)

기사 위치는 REST API가 아닌 **Supabase Realtime Broadcast** 채널로 전송하여 지연 최소화.

```javascript
// 프론트엔드 (React PWA) - 위치 전송
const channel = supabase.channel('driver-location')

// 5초 간격으로 GPS 위치 브로드캐스트
navigator.geolocation.watchPosition((pos) => {
  channel.send({
    type: 'broadcast',
    event: 'location_update',
    payload: {
      driver_id: 'DRV-123',
      lat: pos.coords.latitude,
      lng: pos.coords.longitude,
      heading: pos.coords.heading,
      speed_kmh: pos.coords.speed * 3.6,
      timestamp: new Date().toISOString()
    }
  })
})
```

```
동시에 DB에도 최신 위치 저장 (Edge Function이 Broadcast를 수신하여 처리):

driver_locations 테이블:
| driver_id | lat | lng | heading | speed_kmh | current_dong_code | updated_at |
→ UPSERT로 항상 최신 1건만 유지
→ PostGIS ST_Contains로 현재 위치 → 행정동 자동 매핑
```

- 경로 추천/재계산 API 호출 시 서버가 `driver_locations` 테이블에서 최신 위치를 자동 조회

### 6.2 경로 추천 요청 (Supabase Edge Function)

```
POST /functions/v1/route-recommend

Request Body:
{
  "driver_id": "DRV-123",      // 서버가 driver_locations 테이블에서 현재 위치/heading 자동 조회
  "orders": [
    {
      "order_id": "ORD-001",
      "pickup": { "lat": 37.5700, "lng": 126.9850 },
      "dropoff": { "lat": 37.5100, "lng": 127.0500 },
      "status": "in_progress"  // in_progress | picked_up | pending
    }
  ],
  "max_recommendations": 3     // 추천 경로 최대 개수 (기본 3)
}
```

- `current_location`, `heading_direction`은 별도 전송 불요 → Edge Function이 `driver_locations` 테이블에서 자동 조회

## 7. 데이터 파이프라인

### 7.1 초기 데이터 구축

```
[공공 API 수집] → [서울 행정동별 시설 집계] → [콜 기대지수 산출] → [Supabase DB 적재]

1. Supabase Edge Function (또는 로컬 스크립트)로 소상공인 상가정보 API 서울 데이터 수집
2. 각 사업체의 좌표 → PostGIS ST_Contains로 행정동 매핑
3. 행정동별 카테고리별 시설 수 집계
4. 스코어 공식 적용하여 행정동 콜 기대지수 산출
5. 행정동 폴리곤 기반 인접동 매핑 테이블 생성
6. Supabase PostgreSQL + PostGIS에 적재
```

### 7.2 정기 갱신 (배치)
- 주 1회: pg_cron 또는 Supabase Edge Function(Cron Trigger)로 공공 API 재수집 → 스코어 재계산

## 9. 주요 고려사항

### 9.1 경로 품질
- 우회 제한을 두지 않되, 각 추천 경로마다 최단경로 대비 추가 시간을 명확히 표시하여 기사가 판단
- 추천 경로 정렬 기준: 콜 기대지수 합산 높은 순 (기사가 +시간과 콜 기대치를 비교하여 선택)
- 콜 기대지수가 낮은 지역만 있을 경우 무의미한 추천 방지 → 최소 스코어 임계값 이하 행정동은 경유 후보에서 제외

### 9.2 성능
- 행정동 스코어는 사전 계산 + DB 인덱스 최적화 → 경로 계산 시 쿼리 부하 최소화
- 인접동 매핑도 사전 계산 테이블 → 실시간 공간 쿼리 최소화
- Supabase Edge Function cold start 고려 → 자주 호출되는 함수는 warm 유지
- 경로 계산 응답시간 목표: 2초 이내

### 9.3 실시간 위치 추적
- GPS 수신 불량 (터널, 지하) 시 마지막 유효 위치 + heading 기반으로 위치 추정
- 위치 업데이트 누락 시 타임아웃 처리 (예: 30초 이상 미수신 → 위치 불확실 플래그)
- `driver_locations` 테이블의 `updated_at` 기준으로 오래된 위치 판별
- 배터리 소모 최적화: 오더 진행중일 때만 5초 간격, 대기중일 때는 30초 간격
- **PWA 백그라운드 GPS 제약 (중요)**:
  - iOS Safari PWA는 백그라운드 진입 시 GPS 수집이 중단됨 (OS 레벨 제약, 우회 불가)
  - 대응: 네비게이션 진행 중에는 **Screen Wake Lock API**로 화면 꺼짐 방지
  - 앱 진입 시 "화면 켜짐 유지" 권한 안내 팝업 표시, 기사가 수동 허용
  - Wake Lock이 해제되거나 미지원 브라우저일 경우 "화면을 켜두세요" 상단 알림 배너 상시 표시
  - 화면이 꺼져 위치 전송이 중단된 경우, 앱 복귀 시 즉시 현재 위치 재수집 후 경로 갱신

### 9.4 전진 방향 제약
- 실시간 GPS heading + 이미 방문한 포인트 이력 관리
- 신규 오더의 픽업지가 후방인 경우: 전방 우회 경로 탐색, 불가능 시 기사에게 알림

### 9.5 스코어 정확도
- 공공데이터 기반 추정값으로 운영
- A/B 테스트로 스코어 가중치 튜닝
